Waiting for lock
Timeout (5s)
Lock mechanism
Global distributed lock
All subscription submits share a single lock (subscriptionSubmit) using SELECT ... FOR UPDATE on a DB row. Lock hold time is only ~15–30ms.
Unfair polling
When the lock is held, threads sleep 100ms then retry. InnoDB wakes waiters in no particular order — newer threads can "cut in line" ahead of older ones. No FIFO guarantee.
Starvation under load
At 23 req/s with 13 concurrent threads, 9 unlucky threads kept losing the poll race for the full 5s timeout. During each failure, ~50 other threads acquired and released the lock successfully (~20ms each).

Fix: two-level fair lock (commit 5cd440c5b)
1. Local fair lock
ReentrantLock(fair=true) per lock name in a ConcurrentHashMap. Threads within the same JVM queue in strict FIFO order. Only the front-of-queue thread attempts the DB lock.
2. DB distributed lock (unchanged)
Same SELECT FOR UPDATE, but now sees at most 2 concurrent competitors (one per JVM instance).
3. 20ms yield + retry
Prevents one JVM from monopolizing the DB lock. With 2 JVMs and ~50ms amortized throughput, capacity is ~40 locks/sec — well above the 23 req/s peak.

Prevention measures
Alert on any billing.tooFast
A single occurrence should page — it's a leading indicator of lock contention.
Caller-side retry with jitter
Exponential backoff + jitter before aborting a batch. Defense-in-depth against future lock regressions.
Best-effort batch processing
Don't abort an entire org batch on a single-item failure. Continue other items, retry failed ones at the end.
Document lock semantics
LockService Javadoc should state FIFO fairness guarantees and ~20ms DB granularity to prevent future "optimizations" from reintroducing starvation.
